Ice World: Algorithms

Procedural Generation & Simulation Algorithms in Unity

Michael Sheroubi  |  December 2025    Unity Procedural Generation Simulation Reinforcement Learning AI / Boids

🎨 View Full Canva Presentation »

Overview

Ice World is a Unity project built around a suite of interconnected procedural and simulation systems. Each system is a standalone Unity Editor tool or runtime component — covering village layout generation, icicle and castle mesh synthesis, fractal snowflakes, GPU/CPU synchronized Gerstner wave shaders, a classic boids flocking simulation, a Q-learning wolf pack hunter, species-specific prey AI, procedural ski path generation, and a physics-driven snowball carver. The emphasis throughout is on clean algorithmic design: discrete grid models, recursive fractals, mesh batching for render performance, and mathematical synchronization between GPU shaders and CPU physics.

1. Village Generator

The VillageGenerator script is a Unity Editor tool that procedurally generates an organic medieval-style village on a grid, including a town square, wandering roads, and irregular houses that hug nearby streets. The algorithm combines a discrete layout phase on a 2D grid with mesh batching to produce a single optimized prefab.

Grid Structure & Parameterization

The generator operates on a gridSize × gridSize square grid where each cell represents a world-space tile of size tileSize. Each tile is assigned a type from {Empty, Road, House, Square}, stored in grid[x,z]. Key parameters:

  • Layout: gridSize, tileSize, townSquareSize
  • Organic factors: irregularity, rotationChaos, streetCloseness
  • Architecture: houseDensity, minHeight, maxHeight, roofHeight

Generation Pipeline

1
Grid layout: Allocate the grid, call GenerateOrganicLayout() to place the town square and wandering roads, then PlaceHouses() to fill adjacent empty cells stochastically.
2
Mesh accumulation: Iterate over all non-empty tiles — Square/Road tiles become flat scaled planes; House tiles compute organic position/rotation and call BuildHouse() to produce wall + roof mesh entries.
3
Final assembly: Each category list (roadMesh, squareMesh, wallMesh, roofMesh) is combined via CombineInstance, attached as children under a village root, and saved as a prefab.

Town Square & Wandering Roads

The town square is placed slightly off-center (random offset in [−5, 5] cells, clamped to bounds) and filled with TileType.Square. Roads are generated via multiple biased random walks from the town square:

Start: (x, z) at square center, direction (dirX, dirZ) ∈ {-1,0,1}² \ {(0,0)}
Each step: mark cell as Road if valid & empty
With P=0.3: perturb dirX or dirZ (meander).
Move to (x+dirX, z+dirZ); stop at boundary.

This guarantees contiguous road paths while producing organic, non-grid-aligned trajectories.

House Placement & Street Hugging

Adjacency-Based Seeding

For each interior cell (x, z) that is Empty and has at least one 4-neighbor that is Road or Square, the tile becomes House with probability equal to houseDensity. Houses only appear next to streets, forming street-facing rows.

Organic Pose Computation

The final house transform is computed in four steps:

1. Base position: p_base = (x·tileSize, 0, z·tileSize) + offset
2. Street hugging: p_street = p_base + d⃗ · (0.4 · tileSize · streetCloseness)
3. Positional noise: Δx,Δz ~ U(-irregularity·tileSize, +irregularity·tileSize)
p_final = p_street + (Δx, 0, Δz)
4. Rotation: R_face = LookRotation(-d⃗)
θ ~ U(-rotationChaos, +rotationChaos)
R_final = R_face · Ry(θ)

This yields buildings that visually align to streets but are imperfectly placed and rotated, producing an organic hand-built feeling.

House Geometry

Walls: A cube primitive is scaled to random width and depth:

w ~ tileSize · U(0.6, 0.9), d ~ tileSize · U(0.6, 0.9), h ~ U(minHeight, maxHeight)

Gabled Roof: A custom triangular prism from GeneratePrismMesh. Two triangular gables (front/back) and two sloped roof faces are defined using half-width hw = w/2 and half-depth hd = d/2. The apex ridge runs along the house length at height roofHeight. The roof sits at p_final + (0, h, 0) with R_final.

Roads, Squares & Mesh Batching

For each road/square tile, AddFlatTile instantiates a Unity Plane, scales it by tileSize · 0.105 (slight overlap to avoid gaps), adds its mesh to the appropriate CombineInstance list, and immediately destroys the temporary object. All four categories (Roads, TownSquare, Walls, Roofs) are then combined into large static meshes to minimise draw calls, and the result is saved as a reusable prefab.

2. Icicle Generator

The IcicleGenerator script procedurally generates 3D icicle clusters and saves them as mesh and prefab assets. It constructs tapered cylindrical meshes with vertex irregularities, then batches them into a single optimized mesh.

Configuration Parameters

GroupParameterEffect
ClustericicleCount, clusterSizeNumber of icicles and width/depth area of the cluster
ShapeminLength, maxLength, baseRadiusLength range and base width of each icicle
DetailradialSegments, verticalSegments, irregularityMesh resolution and surface wobble
OutputsavePathDestination for generated mesh and prefab assets

Cluster Generation Pipeline

For each icicle, a position is sampled within the cluster area:

x ~ U(-W/2, W/2), z ~ U(-D/2, D/2), L ~ U(minLength, maxLength)

Each icicle is built with CreateProceduralIcicle(L), combined via a local TRS matrix, then all instances are merged into one mesh.

Procedural Icicle Mesh

Vertices are generated in concentric rings along the vertical axis using polar coordinates with linear tapering and noise:

θ_r = 2π·r / R_s
t_k = k / V_s
R_k = (1 - t_k)·R₀ (linear taper from base to tip)

v_{r,k} = (cos(θ_r)·R_k + Δx, −t_k·L, sin(θ_r)·R_k + Δz) for body rings
v_tip = (0, −L, 0) for the apex

Δx, Δz ~ U(−irregularity, +irregularity)

Adjacent rings are connected with quad strips; the final segment fans into the shared tip vertex forming a conic point. The combined mesh is saved as a timestamped asset and wrapped in a prefab with MeshFilter + MeshRenderer.

3. Castle Generator

The CastleGenerator builds a rectangular castle from four cylindrical corner towers, four connecting wall segments, and optional battlements — all combined into a single mesh prefab with a BoxCollider.

Layout Parameters

  • Footprint: width, length, wallHeight, wallThickness
  • Towers: towerHeight, towerRadius, towerResolution (roundness)
  • Details: addBattlements, battlementSize

Generation Pipeline

1
Setup: Ensure save directory exists; initialize CombineInstance list.
2
Corner towers: Compute half-dimensions w_h = width/2, l_h = length/2. Place a cylinder at each of the four corners (±w_h, 0, ±l_h), scaled to (2·towerRadius, towerHeight/2, 2·towerRadius). Optionally ring the top with 8 battlements.
3
Walls: Front/back walls centered at (0, 0, ±l_h) with length reduced by 1.5·towerRadius on each side to avoid overlapping towers. Left/right walls similarly centered at (±w_h, 0, 0). Each wall optionally crowned with linear crenellations.
4
Mesh combination & prefab export: All instances merged into a 32-bit index mesh; normals recalculated; prefab created with MeshFilter, MeshRenderer, and BoxCollider.

Battlement Geometry

Ring Battlements (Towers)

8 cubic blocks distributed around the tower top circumference:

θ_i = i · 360° / 8 (i = 0..7)
o⃗_i = Ry(θ_i) · forward⃗ · towerRadius
p_i = (x, towerHeight, z) + o⃗_i + (0, battlementSize/2, 0)

Linear Battlements (Walls)

Starting from one end of the wall top, blocks are placed at alternating "block" and "gap" intervals of battlementSize, producing a crenellated pattern. For horizontal walls:

p = p₀ + (currentDist + battlementSize/2, battlementSize/2, 0)
scale = (battlementSize, battlementSize, wallThickness)

Vertical walls use the Z axis instead. A boolean flag alternates the gap pattern each step.

4. Snowflake Generator

The SnowflakeGenerator produces a 3D fractal snowflake by recursively branching segments around six radial arms, with geometric length decay and angle offsets. All segment meshes are combined into a single optimized asset.

Parameters

  • recursionDepth — levels of branching
  • initialLength — base arm length
  • widthMultiplier — segment thickness relative to length
  • lengthDecay — shrinkage per recursion level
  • branchAngle — angular offset of lateral branches

Six-Fold Symmetry

The same recursive branching rule is applied six times, once per arm, each rotated by 60°:

R_i = Rz(60° · i) for i ∈ {0, 1, 2, 3, 4, 5}
GenerateBranch(origin, R_i, initialLength, recursionDepth)

Recursive Branching Rule

Stopping condition: currentDepth ≤ 0 → return

For each call, a rectangular prism segment is placed along the branch direction:

w = length · widthMultiplier
p_end = p_start + R · up⃗ · length
AddSegment(p_start, p_end, w, R)

Three recursive children are then spawned:

p_branch = Lerp(p_start, p_end, 0.6)
L' = length · lengthDecay

R_left = R · Rz(+branchAngle)
R_right = R · Rz(-branchAngle)

GenerateBranch(p_branch, R_left, L', depth-1) ← left side branch
GenerateBranch(p_branch, R_right, L', depth-1) ← right side branch
GenerateBranch(p_end, R, 0.5·L', depth-1) ← shortened stem continuation

Segment Geometry

Each branch segment is a scaled Unity cube (rectangular prism). The cube is sized to (w, L, 0.2w), positioned at the segment midpoint, and rotated to align its local up axis with the branch direction. A CombineInstance is filled and the temporary object destroyed. After all recursion, the final combined mesh is saved as a 32-bit indexed asset and optionally packed into a prefab.

5. Wave Shaders — Gerstner Waves

Standard sinusoidal waves produce symmetric peaks and troughs. Real ocean waves exhibit sharp crests, flat troughs, and circular orbital motion. Gerstner waves solve this by displacing vertices in circular orbits — producing peaked crests and a convincing rolling swell.

The Gerstner Wave Displacement

P(t) = P₀ + ΔP(t)

ΔP(t) = | A · D_x · cos(f) | (X displacement)
| A · sin(f) | (Y displacement — height)
| A · D_z · cos(f) | (Z displacement)

f = k · (D⃗ · P_xz − c·t) (phase function)

k = 2π/λ (wave number)
c = √(g/k) (wave speed, g ≈ 9.8 m/s²)
A = steepness (amplitude)

Vertices rise and fall (Y via sin(f)) while also moving horizontally in the wave direction (X/Z via cos(f)), producing orbital motion.

WaveManager — Singleton & Global State

WaveManager is a singleton that provides a single source of truth for all wave parameters, accessible globally throughout the game. It uses DontDestroyOnLoad to persist across scenes.

ParameterTypeMeaning
wavelengthfloatWavelength λ in world units. Larger λ = slower, broader waves.
steepnessfloatAmplitude A ∈ [0,1]. Controls wave height.
speedfloatMultiplier for wave speed c. Adjusts temporal phase evolution.
directionVector2Normalized direction (D_x, D_z) of wave propagation.

Every frame, parameters are broadcast to the GPU via global shader properties:

private void Update() { Shader.SetGlobalFloat("_WaterWavelength", wavelength); Shader.SetGlobalFloat("_WaterSteepness", steepness); Shader.SetGlobalFloat("_WaterSpeed", speed); Shader.SetGlobalVector("_WaterDirection", direction); }

CPU Buoyancy — GetWaveHeight

The vertical component of the Gerstner wave is evaluated in C# so physics objects see the same wave field as the shader:

y = A · sin(k · (D⃗ · P_xz − c·t))
public float GetWaveHeight(Vector3 position) { float k = 2 * Mathf.PI / wavelength; float c = Mathf.Sqrt(9.8f / k); Vector2 d = direction.normalized; float f = k * (Vector2.Dot(d, new Vector2(position.x, position.z)) - c * Time.time * speed); return steepness * Mathf.Sin(f); }
Only Y is computed — the full Gerstner implementation displaces X and Z too, but computing only the vertical component is sufficient for buoyancy physics and is significantly cheaper. The X/Z orbital motion is handled purely by the GPU shader.

GPU Shader Graph Vertex Displacement

The Shader Graph reads global wave parameters and applies the full displacement to every mesh vertex:

P_new = P_old + steepness · | D_x · sin(f) |
| sin(f) |
| D_z · sin(f) |

GPU/CPU Synchronization — Why It Matters

Because both the GPU shader and the CPU physics use the same WaveManager parameters and evaluate equivalent math:

  • When wavelength changes, both shader and physics immediately update.
  • When steepness increases, waves appear taller visually and exert stronger buoyancy.
  • When direction rotates, the entire wave field rotates consistently on both GPU and CPU.
Without synchronization, visual waves drift ahead of physics waves, causing floating objects to sink through or float above the rendered surface — a jarring artifact.

6. Boids Flocking Simulation

A classic boids simulation with separation, alignment, and cohesion, constrained to a cylindrical "flight deck" volume and extended with raycast-based terrain avoidance.

Configuration

  • Spawn: boidPrefab, spawnCount, spawnRadius
  • Dynamics: minSpeed, maxSpeed, perceptionRadius
  • Weights: separationWeight, alignmentWeight, cohesionWeight
  • Cylindrical bounds: boundsRadius, minHeight, maxHeight, boundsWeight
  • Obstacle avoidance: obstacleMask, obstacleAvoidDistance

Spawning

Each boid is placed randomly in a vertical cylinder, with initial velocity random in XZ:

(x,z) from Random.insideUnitCircle · spawnRadius
y ~ U(minHeight, maxHeight)
v₀ = Random.insideUnitSphere · maxSpeed, v₀.y = 0

Neighborhood-Based Flocking Forces

For each boid, three steering forces are computed over neighbors within perceptionRadius:

Separation: F_sep += (p - p_other) / d² (inverse-square repulsion)
Alignment: F_align = (1/N)·Σ v_other − v (match average heading)
Cohesion: F_coh = (1/N)·Σ p_other − p (attract to centre of mass)

Cylindrical Bounds Enforcement

o⃗ = (p_x − c_x, 0, p_z − c_z)
if |o⃗| > boundsRadius: F_bounds += −ô (inward horizontal)
if p.y < minHeight: F_bounds += 5·ŷ (push up)
if p.y > maxHeight: F_bounds += -5·ŷ (push down)

Obstacle Avoidance via Raycasts

  • Forward ray: cast along v̂ up to obstacleAvoidDistance. On hit: F_obs = Reflect(v⃗, n̂) · 10
  • Ground ray: cast straight down. On hit: F_obs += 10·ŷ

Force Integration & Speed Clamping

F_total = F_sep·w_sep + F_align·w_align + F_coh·w_coh + F_bounds·w_bounds + F_obs
v ← v + F_total · Δt

if |v| > maxSpeed: v = v̂ · maxSpeed
if |v| < minSpeed: v = v̂ · minSpeed

p ← p + v · Δt
R ← Slerp(R, LookRotation(v⃗), 5·Δt) (smooth facing)
The per-frame update is O(N²) in boid count — straightforward and visual, but not optimized for very large flocks (spatial hashing or compute shaders would be needed beyond ~500 agents).

7. Hunt: Wolf Pack Q-Learning & Prey AI

Wolf Pack Brain — Tabular Q-Learning

The PackBrain class implements tabular Q-Learning, mapping environmental states (prey type) to effective actions (hunting tactics).

State-Action Space

DimensionValues
State S — prey typeRabbit, Bear, Goat
Action A — formationSurround (0) · FlankV (1) · Blitz (2)

Q-Update Rule

Q_new(S,A) ← Q(S,A) + α · (R − Q(S,A))

α = 0.5 (learning rate)

Reward Function

R = +20 if hunt successful
= −5 if hunt failed (time/energy wasted)
= −100 · N_dead penalty per wolf lost

ε-Greedy Decision Policy

With probability ε, a random tactic is chosen (exploration); otherwise the highest Q(S,A) tactic is selected (exploitation). ε decays over time:

ε ← max(0.1, ε · 0.99)

Formation Geometry

Surround Tactic

P_slot,i = P_prey + RotateY(360°/N · i) · F_forward · r

FlankV Tactic

P_slot,i = P_prey − (F_prey · (r + d_i)) + (R_prey · s_i · (0.5r + d_i))
d_i = depth, s_i = ±1 alternating side

Greedy Slot Assignment (Prevent Path Crossing)

s* = argmin_{s∈S} |P_w − P_s| (assign nearest slot to each wolf, remove when claimed)

Hunt Phases

  • Preparation (18 m radius): Wolves move to preparation slots and wait until 80% of the pack has assembled.
  • Attack (2.5 m kill distance): Target radius collapses; wolves converge on the prey.

Individual Wolf — SettleDown Noise

To avoid robotic crowd synchronization, a coroutine desynchronizes each wolf during the preparation phase:

private IEnumerator SettleDownRoutine() { isSettling = true; // 1. Random delay to desynchronize pack float initialWait = Random.Range(1.0f, 4.0f); yield return new WaitForSeconds(initialWait); // 2. "Circling" noise: move to a random offset Vector3 randomAdj = transform.position + (Random.insideUnitSphere * 1.5f); if (NavMesh.SamplePosition(randomAdj, out hit, 2f, NavMesh.AllAreas)) agent.SetDestination(hit.position); // 3. Final micro-wait before sitting yield return new WaitForSeconds(Random.Range(0.5f, 1.5f)); SitDown(); }

Prey AI — SmartPrey

Prey sense predators via a spherical detection zone + field-of-view raycast check. On detection, behavior branches by species:

SpeciesBehavior
RabbitMulti-ray sensor (front/back/side/diagonal). If ≥3 rays detect predators → panic mode at panicSpeed; otherwise flee via FleeFrom
GoatAlways flees at goatFleeSpeed using FleeFrom without extra panic logic
BearStops moving; smoothly rotates to face predator — preparing for confrontation/attack

Fleeing computes a direction away from the predator, proposes a point along that vector, validates via NavMesh.SamplePosition, and sets the agent destination.

8. Ski Path Generation

For each requested trajectory T_i, a skeleton path is built from a start position using terrain-following fall-line analysis and a sine-modulated turn function, then smoothed with a Catmull-Rom spline.

Skeleton Path — Step Loop

For each step k from 0 to S:

A. Surface Analysis — The Fall Line

A ray is cast downward; the terrain normal n is retrieved and the fall line computed:

v_fall = normalize(ProjectOnPlane(v_down, n))

B. Turn Logic — S-Turns

θ_turn = A · sin(ω · k + φ_rand)

A = Turn Amplitude (width of turn)
ω = Turn Frequency (tightness)
φ_rand = random phase offset per trajectory

The target direction v_target is obtained by rotating v_fall around Y by θ_turn.

C. Next Waypoint

p_flat = p_k + (v_target · d_step)
→ raycast from above p_flat downward → p_ground

D. Obstacle Validation

A linecast checks from p_k to p_ground on ObstacleLayer. If clear: p_{k+1} = p_ground. If blocked: path terminates at p_k.

Path Smoothing — Catmull-Rom Spline

The raw waypoint skeleton is smoothed between each pair of consecutive points P_1, P_2 using four control points:

P(t) = 0.5 · [1, t, t², t³] · | 0 2 0 0 | · [P₀] | -1 0 1 0 | [P₁] | 2 -5 4 -1 | [P₂] | -1 3 -3 1 | [P₃]

Each segment is subdivided into M sub-points (smoothSubdivisions) to produce a high-resolution, fluid trajectory.

Complexity

MetricValue
Time complexityO(N · S · M) — trajectories × steps × subdivisions
Physics cost≈ 3 · N · S queries (2 raycasts + 1 linecast per step)

9. Snowball Carver

The SnowballCarver script governs a physical snowball that dynamically grows as it rolls and carves smooth depressions into Unity terrain through a deferred command-queue system.

Dynamic Growth Physics

While rolling (velocity above minVelocity) and below maxScale, scale and mass increase each frame:

ΔS = d · r · Δt

d = distance moved this frame
r = growthRate
Δt = Time.deltaTime

ΔS is applied uniformly to X, Y, and Z scale. Mass increases proportionally to maintain physical weightiness.

Deferred Carving — Command Queue

Terrain is not modified immediately. Instead, a CarveCommand is enqueued when both conditions are met:

  1. Velocity threshold: the snowball is rolling (isMovingFastEnough)
  2. Spatial sampling: horizontal distance since last sample > distanceBetweenCarves

Each command captures: position, radiusScale, and executeTime = Time.time + carveDelaySeconds.

Every frame, ProcessCarveQueue peeks at the front. If Time.time ≥ executeTime, the command is dequeued and applied.

Terrain Deformation Mathematics

Coordinate Mapping

x_hm = ⌊ (P_world.x − T_pos.x) / T_size.x · R_width ⌋
(analogously for z_hm)

Spherical Cosine Falloff

For every heightmap pixel within dynamicRadius, a smooth bowl-shaped depression is applied:

H_new = H_old − (Depth · cos(dist/R · π/2))

This creates a smooth impression rather than a jagged hole. SetHeightsDelayLOD is used to efficiently update both the physics collider and the visual mesh.

The deferred queue is essential: applying terrain modifications immediately each frame would cause performance spikes and produce a "drilling" artifact when the snowball is stationary. The spatial sampling threshold ensures the trail is sampled at consistent intervals regardless of framerate.
