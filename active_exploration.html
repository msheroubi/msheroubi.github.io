Active 3D Scene Exploration with a Robotic Arm

Iterative Reconstruction, Occupancy Mapping, and Learned Viewpoint Selection

Michael Sheroubi  |  March 2026    Robotics Computer Vision Reinforcement Learning 3D Reconstruction

Abstract

A complete pipeline for autonomous 3D scene exploration using a 6-DOF robotic arm equipped with a wrist-mounted camera. The system iteratively captures images, reconstructs the scene using the VGGT foundation model, builds a tri-state occupancy grid via 3D DDA ray-casting, and selects the next-best viewpoint to maximise coverage of unknown regions. A fixed-size occupancy grid ensures consistent voxel resolution across reconstruction rounds. A reinforcement learning approach using PPO with a 3D-CNN policy was also developed, trained entirely in an analytic ray-casting simulator, enabling viewpoint selection without heuristics. The system operates in a split architecture: the robot arm and camera run on a local machine, while GPU-intensive reconstruction runs on a remote server connected via WebSocket.

1. Introduction

Mapping an unknown environment from a fixed-base robotic arm poses a fundamental challenge: the camera can only observe a fraction of the scene from any single viewpoint, and the arm's workspace is constrained by its kinematic limits. The system must therefore decide where to look next to maximise the information gained per movement.

This project tackles the next-best-view (NBV) problem in a tabletop 3D reconstruction setup:

  • A LeRobot SO-101 arm (6-DOF) sits in front of an unknown object or scene.
  • A wrist-mounted USB camera (640×480) captures RGB images.
  • VGGT (Visual Geometry Grounded Transformer) reconstructs dense 3D geometry from captured views.
  • A tri-state occupancy grid tracks which parts of the scene have been observed.
  • A planner selects viewpoints that maximise the number of newly observed voxels.

The pipeline runs in a client–server architecture: the arm and camera are connected to a local machine, while VGGT runs on a GPU server. Communication uses WebSocket for real-time streaming and REST for reconstruction triggers.

Why VGGT over Depth Anything V2? Depth Anything V2 (DA2) produced cleaner, smoother depth maps with fewer artefacts. However, DA2 only estimates monocular depth — it does not predict camera poses or intrinsics. VGGT jointly estimates camera poses, intrinsics, depth, and 3D point maps in a single forward pass, eliminating the need for a separate SfM pipeline like COLMAP.

Local Machine

active_capture.py is the local controller which:

  1. Connects to the SO-101 arm via serial (LeRobot SDK).
  2. Reads the wrist camera at 640×480.
  3. Computes the camera-to-world transform via forward kinematics (FK) for each joint configuration.
  4. Encodes each frame as JPEG alongside the 7-DOF pose (x, y, z, qx, qy, qz, qw) and streams it over WebSocket.
  5. After reconstruction, receives next-best-view suggestions and moves the arm accordingly.

GPU Server

The server (server.py) exposes a FastAPI application managing:

  • Session management — Start/stop frame collection, enforcing a minimum baseline between accepted frames.
  • Reconstruction — Run VGGT on the collected frames, producing point clouds, depth maps, and occupancy grids.
  • Planning — Compute next-best-view commands from the occupancy grid.

3. The SO-101 Robotic Arm

The LeRobot SO-101 is a low-cost, open-source 6-DOF robotic arm. The camera is mounted to the left of the gripper and angled ≈5° to the right so that its optical axis converges with the gripper's pointing direction.

Joint Name Limits (deg) Role

J₁Shoulder[−90, 0]Elevation control
J₂Elbow[+20, +110]Reach / fold
J₃Wrist pitch[−60, +10]Tilt fine-tuning
J₄Wrist roll[−90, +90]Rotation about optical axis
J₅Gripper[0, +50]Fixed at 50° during scanning

Viewpoint Bank Generation

Rather than using inverse kinematics (unavailable for this arm), a dense bank of candidate viewpoints is pre-generated by sampling joint angles using a 5-dimensional Halton quasi-random sequence within conservative joint limits. Self-collision-prone configurations are filtered using heuristics (e.g., shoulder + elbow too folded). The default bank contains ~200 viewpoints supplemented by 16 hand-tuned defaults.

The FK function ee_pose(joints) returns a 4×4 camera-to-world matrix that already accounts for the camera's physical offset from the gripper tip.

4. 3D Reconstruction with VGGT

Model Overview

VGGT (Visual Geometry Grounded Transformer, Facebook Research) is a feed-forward foundation model that takes S RGB images and jointly predicts:

  • Camera extrinsics Ei ∈ ℝ3×4 — rotation + translation
  • Camera intrinsics Ki ∈ ℝ3×3 — focal lengths, principal point
  • Depth maps Di ∈ ℝH×W with per-pixel confidence
  • World-space point maps Pi ∈ ℝH×W×3 with per-pixel confidence

Images are resized to width 518 (VGGT's expected input), with height adjusted to the nearest multiple of 14 (vision transformer patch size). For 640×480 input, the effective resolution is 518×392.

Point Cloud Fusion

After VGGT's forward pass, per-view point maps are fused into a single point cloud by:

  1. Filtering pixels with confidence below a threshold (default 0.3).
  2. Concatenating all surviving 3D points across views.
  3. Associating each point with its RGB colour from the input image.
VGGT vs. Depth Anything V2: DA2 produced cleaner depth maps, but requires a separate SfM pipeline for pose estimation. VGGT's joint prediction takes only 3–10 seconds for 10–50 frames, making it better suited for the iterative exploration loop where latency matters.

5. Occupancy Grid

The occupancy grid is a regular 3D voxel grid of resolution G³ (default G=64, yielding 262,144 voxels). Each voxel has one of three states:

ValueStateMeaning
0 UNKNOWN Never observed by any camera ray.
1 FREE At least one ray passed through without hitting a surface — confirmed empty.
2 OCCUPIED A ray terminated here — surface point detected.

The exploration fraction is defined as:

f_explored = ( |{FREE}| + |{OCCUPIED}| ) / G³

The exploration loop terminates when f_explored ≥ 0.90.

3D DDA Ray-Casting (Amanatides & Woo)

Each voxel's state is determined by ray-casting from camera origins through the voxel grid to observed surface points. The 3D Digital Differential Analyser (DDA) algorithm efficiently traverses voxels along a ray by tracking the parametric distances to the next axis-aligned voxel boundaries.

Algorithm: 3D DDA Ray Traversal
1Input: Camera origin o, surface endpoint e, grid origin, voxel size, grid G³
2d ← normalise(e − o)
3(i_x, i_y, i_z) ← ⌊o / voxel_size⌋ // Starting voxel
4(e_x, e_y, e_z) ← ⌊e / voxel_size⌋ // Endpoint voxel
5Compute step_xyz, t_max_xyz, Δt_xyz for each axis
6while voxel is inside grid:
7   if (i_x, i_y, i_z) == (e_x, e_y, e_z):
8      grid[i_x, i_y, i_z] ← OCCUPIED
9      break
10  else:
11     if grid[i_x, i_y, i_z] ≠ OCCUPIED:
12        grid[i_x, i_y, i_z] ← FREE
13   Advance to next voxel along smallest t_max axis

For each reconstructed view, rays are cast at every 4th pixel (ray_stride=4), resulting in ~12,740 rays per view (130×98). Crucially, existing OCCUPIED markings are never overwritten by FREE, preserving previously detected surfaces.

The DDA implementation includes an optional Numba JIT-compiled path that parallelises ray-casting across all rays, providing a ~25× speedup over the pure-Python fallback.

Fixed-Size Occupancy Grid

The initial implementation used a dynamic bounding box computed from the point cloud extent, which caused three critical problems:

  1. Voxel size changed between rounds — a voxel marked FREE in round k might correspond to a different physical region in round k+1.
  2. Exploration fraction became meaningless — f_explored could decrease as new points expanded the bounding box, making the stopping criterion unreachable.
  3. Planner instability — frontier locations jumped around, leading to oscillating viewpoint suggestions.

Solution: The grid's bounding box is fixed at the start of exploration using the camera's FK from a known reference pose. The scene centre is placed 25 cm in front of the camera along its optical axis:

c_scene = p_cam + 0.25 · d̂_cam
o_grid = c_scene − h · 1
v_size = 2h / G

Hemisphere Masking

Because the arm sits on one side of the scene, the rear half of the bounding cube can never be observed. Voxels behind the scene centre are pre-marked as FREE, preventing the planner from targeting unreachable regions and accelerating convergence of f_explored.

6. Next-Best-View Planning

Heuristic Planner

The baseline planner operates in two stages:

Stage 1 — Frontier Detection

Frontier voxels are UNKNOWN voxels with at least one 6-connected FREE neighbour. They represent the boundary of explored space — the most promising targets for new observations. Frontier centroids are clustered using non-maximum suppression with a radius of 2 voxels.

Stage 2 — Information-Gain Scoring

For each unvisited viewpoint candidate, an information-gain score counts UNKNOWN voxels within a frustum cone defined by:

  • Half-angle: 0.52 rad (~30°, matching the camera's ~60° horizontal FOV)
  • Maximum range: 0.5 m

Viewpoints closer to the scene centre receive a proximity bonus. The top-k candidates are then reordered via a nearest-neighbour TSP heuristic in joint space to minimise arm travel.

Exploration Loop

Algorithm: Automatic NBV Exploration Loop
1Initialise scene bounds from reference pose
2Generate viewpoint bank 𝒱 (Halton sampling, |𝒱| ≈ 200)
3Pre-compute FK positions and directions for all candidates
4𝒷₀ ← seed views (first 3–5 from defaults)
5for round k = 1, 2, …:
6   Move arm to each v ∈ 𝒷_k, capture and stream frames
7   Trigger VGGT reconstruction (merging all sessions)
8   Fetch occupancy grid and exploration fraction
9   if f_explored ≥ 0.90: stop // Scene sufficiently mapped
10  Score unvisited candidates by information gain + NBV matching
11  𝒷_{k+1} ← top-k candidates, TSP-reordered

7. Reinforcement Learning for Viewpoint Selection

While the heuristic planner works well in practice, it does not account for occlusion, uses a simplified cone frustum, and greedily maximises single-step gain. An alternative RL-based viewpoint selection approach was also developed.

Problem Formulation (MDP)

ComponentDescription
State Downsampled occupancy grid (16³), current camera position and direction, exploration fraction
Action 4-dimensional continuous vector (x, y, z, roll) ∈ [−1, 1]⁴, mapped to the nearest viewpoint bank candidate
Reward r_t = f_explored(t) − f_explored(t−1) — the fraction of newly explored voxels
Terminal f_explored ≥ 0.95 or step limit reached (default 20)

  • Encoder — Three Conv3D layers (1→16→32→64 channels) with BatchNorm, ReLU, and max-pooling, followed by adaptive average pooling to a 64-dim vector.
  • Trunk — Concatenation with camera position (3), camera direction (3), and explored fraction (1), followed by a linear layer to 128 dimensions.
  • Actor head — Outputs a 4-dim mean μ (tanh-squashed) and a learned log σ parameter for a diagonal Gaussian policy.
  • Critic head — Outputs a scalar state value V(s).

Total parameters: ~96,000.

PPO Training

  • Generalised Advantage Estimation (GAE, γ=0.99, λ=0.95)
  • Clipped surrogate objective (ε = 0.2)
  • Entropy bonus (β = 0.01–0.02) to encourage exploration
  • Adam optimiser, learning rate 3×10⁻⁴
  • 512 environment steps per epoch, 4 PPO mini-batch passes

Analytic Visibility Simulator

To train the RL policy without requiring the physical arm, a fully analytic simulator was built that generates random scenes and simulates camera observations:

  • Scene generation — Each episode generates 3–8 random primitive objects (boxes, spheres, cylinders) within the inner 60% of the voxel volume, plus a floor plane in the bottom voxel layers.
  • Ray-casting with occlusion — The same 3D DDA algorithm is used, but rays are cast to max_range and checked against the ground-truth grid. Objects block the camera's view of regions behind them, forcing the policy to learn multi-view strategies.
  • Performance — With Numba JIT compilation, the simulator processes ~12,740 rays per observation in <5 ms, enabling ~0.5 s per training epoch (512 steps) on a single GPU node.

8. Critical Analysis: RL vs. Exact Ray-Cast Scoring

A central question of this project: is reinforcement learning the right tool for viewpoint selection in this setting?

What RL Theoretically Adds

  1. Occlusion awareness — The cone test counts all UNKNOWN voxels in the frustum regardless of whether they are occluded by intervening OCCUPIED voxels. A learned policy observing the full occupancy grid could implicitly account for occlusion.
  2. Multi-step planning — RL optimises discounted cumulative reward, so it could learn strategies like "move to a less informative viewpoint now because it enables a highly informative one next."
  3. Learned shape priors — If trained on realistic scenes, the policy could learn that objects have backsides, concavities require specific viewing angles, etc.

Why These Advantages Don't Materialise

Occlusion can be computed exactly. Rather than learning occlusion awareness, we can compute it directly using the same 3D DDA algorithm — yielding an exact, occlusion-aware score with no training, no approximation error, and zero sim-to-real gap.
Multi-step lookahead is intractable. With ~200 candidate viewpoints and a horizon of 20 steps, the decision tree has 200²⁰ branches. PPO with ~96,000 parameters and ~50,000 training steps explores a negligible fraction. In practice, the learned policy converges to a greedy strategy — the very function we can compute analytically.
Random primitives offer no learnable prior. Each training episode generates a new random scene. There is no consistent geometric prior for the policy to exploit. The optimal strategy in every episode reduces to: look where the most UNKNOWN voxels are visible — which is exactly what the exact scorer computes.

The Exact Ray-Cast Scorer

An improved heuristic replaces the simplified cone test with exact DDA ray-casting against the live occupancy grid. For each candidate viewpoint:

  1. Generate a grid of 130×98 ray directions matching the real camera's 60°×47° FOV.
  2. For each ray, walk the DDA through the current occupancy grid to max_range.
  3. If the ray hits an OCCUPIED voxel, stop (occlusion modelled).
  4. Count the UNKNOWN voxels traversed before stopping.
  5. Accumulate across all rays to obtain the score.
With Numba JIT, scoring all ~200 candidates takes <1 s on a single CPU core (~2.5M rays, each traversing ~50 voxels on average).

When Would RL Be the Right Choice?

Despite its limitations in this setting, RL is well-motivated when:

  1. No explicit occupancy grid — If the reconstruction model outputs only point clouds or implicit representations (e.g., NeRFs), an RL policy operating on raw images or latent features can still learn to explore.
  2. Continuous action spaces with IK — If inverse kinematics is available, the policy can output smooth end-effector trajectories enabling finer-grained viewpoint control.
  3. Task-driven exploration — If the objective is semantic (e.g., "find the red cup," "inspect the weld seam"), reward encodes task-specific success criteria that voxel counting cannot capture.
  4. Large-scale learned priors — Trained on thousands of realistic scenes, the policy can amortise planning across the scene distribution, making decisions in constant time regardless of grid resolution.
  5. Dynamic environments — If objects move during exploration, the policy can learn anticipatory strategies that a myopic scorer cannot.
Takeaway: For this specific problem — maximising voxel coverage with a known occupancy grid, a discrete viewpoint bank, and FK-only kinematics — the exact ray-cast scorer is strictly superior. The RL pipeline remains valuable as a research contribution, establishing the infrastructure (environment, simulator, policy architecture) needed for future extensions to task-driven or continuous-action settings.

Communication Protocol

The WebSocket wire protocol sends binary messages from client to server:

[28 bytes: pose (7×float32 LE)] [variable: JPEG bytes]

The server responds with JSON containing real-time analytics (brightness, confidence) and collection status.

Key Design Decisions

  1. No inverse kinematics. The SO-101 SDK provides FK but not IK. All viewpoint selection operates in joint space: a bank of FK-validated candidates is pre-computed, and both heuristic and RL planners select from this discrete bank.
  2. Session merging. Each exploration round starts a new collection session, but reconstruction merges all historical sessions, ensuring the occupancy grid improves cumulatively.
  3. Occupancy downsampling. For the RL policy, the 64³ grid is downsampled to 16³ using a priority-based pooling rule — within each 4³ block, OCCUPIED > FREE > UNKNOWN — preserving structural information while fitting in the CNN's receptive field.

10. Results & Observations

Heuristic Exploration (Physical Arm)

  • The system reaches ~70% explored fraction within 3–4 rounds (15–20 viewpoints).
  • Hemisphere masking and fixed grid ensure monotonic increase in f_explored.
  • TSP joint-space reordering reduces arm travel by ~30% compared to naïve ordering.

RL Training (Simulation)

Training the PPO policy in the analytic simulator for 100 epochs (51,200 environment steps, ~5 minutes):

  • Average episode reward improves from 0.30 to 0.35 (each episode = 10 viewpoints on a new random scene).
  • The policy learns to prefer viewpoints that face large unexplored regions and to avoid redundant views of already-observed areas.
  • The floor plane in the simulator prevents the policy from learning to look through the table.

11. Conclusions & Future Work

This project presents a complete pipeline for active 3D scene exploration combining a robotic arm, feed-forward 3D reconstruction (VGGT), and intelligent viewpoint selection. The fixed-size occupancy grid and hemisphere masking resolved critical issues with dynamic bounding boxes. The analytic visibility simulator provides a fast, geometry-accurate training environment for RL-based planning.

Future Directions

  • Sim-to-real transfer — Fine-tune the RL policy on real-world transitions collected from the arm to close the simulation gap (VGGT noise, calibration error).
  • Multi-object scenes — Extend the simulator to include more complex shapes and materials.
  • Continuous action space — Replace the discrete viewpoint bank matching with direct Cartesian control (requires IK or learned motor primitives).
  • Depth model fusion — Combine VGGT's pose predictions with DA2's cleaner depth maps for higher-quality reconstruction.
